# حالات الطلبات (Order States) — مرجع + أحدث 20 طلب

## جميع الحالات المعرّفة في النظام

### 1. طلبات الصف (Call Center Queue) — `order.status`

| الحالة (EN) | الحالة (عربي) | متى تظهر |
|-------------|----------------|----------|
| `new` | **جديد** | طلب جديد من ERP أو مباشر، لم يتصل به بعد |
| `scheduled` | **مجدول** | تم تحديد موعد اتصال لاحق |
| `confirmed` | **مؤكد** | العميل أكد الطلب؛ بانتظار موافقة القائد أو تحويله لتذكرة |
| `completed` | **منتهي** | تم تحويله لتذكرة (أو انتهى بدون تحويل حسب الإعداد) |
| `canceled` | **ملغى** | تم إلغاء الطلب |

### 2. نوع الخدمة + الحالة المعروضة معاً (نوع الخدمة / الحالة)

| العرض في الواجهة | المعنى |
|------------------|--------|
| **بيع - يحتاج تأكيد** | نوع الخدمة = بيع، والحالة = جديد أو مجدول (لم يُؤكد بعد) |
| **بيع** | بيع + مؤكد |
| **استبدال** | استبدال (replacement) |
| **صيانة** | صيانة (maintenance) |
| **استرجاع** | استرجاع (return) |
| **مؤكد - حدد النوع** | الحالة = مؤكد لكن لم يُحدد نوع الخدمة بعد |
| **مسودة - لم يُحدد النوع** | جديد/مجدول ولم يُحدد النوع |
| **منتهي** / **منتهي · بيع** | الطلب منتهي (مكتمل أو ملغى) |
| **ملغى** | الطلب ملغى |

### 3. حالات Bosta (التوصيل / الشحن)

**مصدر البيانات الفعلي من الـ Backend/MCP:** كل حالة Bosta لكل عميل تُجلب من:

- **Endpoint:** `GET /api/bosta/customer/{phone_number}/orders`
- **MCP:** `hub_mcrm_api_call` مع `operation_id: "bosta_customer_orders"` و `path_params: { phone_number: "01XXXXXXXXX" }`
- **استجابة الـ API:** `{ data: { orders: [...], totalOrders, types }, success }`. كل عنصر في `orders` له:
  - `status.confirmed` (boolean): هل تم تأكيد التسليم من Bosta.
  - `status.timeline` (array): تتابع الأحداث من Bosta (قد يكون فارغاً في الصيغة الموحدة الحالية).
  - `trackingNumber`, `type` (Send / Return to Origin / …), `timestamps.collected`, إلخ.

في الواجهة نعرض تسميات مثل «تم التسليم» عندما `status.confirmed === true` أو عند وجود حدث تسليم في الـ timeline.

| الحالة (EN) | الحالة (عربي) |
|-------------|----------------|
| `delivered` | **تم التسليم** |
| `completed` | **مكتمل** |
| `in_transit` | **قيد الشحن** |
| `out_for_delivery` | **في الطريق للتسليم** |
| `collected` | **تم الاستلام** |
| `sent` | **تم الإرسال** |
| `ready_for_dispatch` | **جاهز للشحن** |
| `in_process` / `processing` | **قيد المعالجة** |
| `pending` | **قيد الانتظار** |
| `confirmed` | **مؤكد** |
| `returned` | **مرتجع** |
| `failed` | **فشل التسليم** |
| `cancelled` | **ملغي** |
| `ready` | **جاهز** |

### 4. تذاكر الخدمة (Hub Tickets) — نفس مفاتيح Bosta أعلاه

تذاكر الاستبدال/الصيانة/الاسترجاع/البيع تستخدم نفس التسميات العربية في الواجهة (قيد الانتظار، مؤكد، قيد المعالجة، تم التسليم، …).

---

## ملخص سريع: "تم تسليم الأوردر" / "تم التوصيل"

- **تم التسليم** = حالة Bosta/تذكرة: `delivered` أو `completed`.
- **تم التوصيل** = نفس المعنى في السياق العملي؛ في الكود تُعرض كـ **تم التسليم** من خلال `getStatusLabel('delivered')` و `getStatusLabel('completed')`.

**مصدر البيانات:**
- الطلبات: `GET /api/call-center/orders?page=1&per_page=20` (٧ مارس ٢٠٢٦).
- حالة Bosta: `GET /api/bosta/customer/{phone}/orders` (MCP: `bosta_customer_orders`، نفس التاريخ). كل صف يعكس حالة Bosta **حسب رقم هاتف العميل** (قد يكون للعميل أكثر من طلب Bosta).

**عناصر الطلب (Items) من ERP:**  
من الـ ERP عند السينك نحصل على **عدد** العناصر فقط (`total_items` / `total_quantity`) ووصف نصي واحد (`shipping_details`، مثل "1 * كبه هفار 2000 وات...") — و**لا** يوجد مصفوفة line items (سطر لكل منتج). الأهم: الـ Backend **لا يخزّن** حالياً `order_description` ولا عناصر الطلب في جدول `orders` عند sync-from-erp؛ الحقول المحفوظة هي: `erp_order_id`, `customer_phone`, `customer_name`, `delivery_address`, `governorate`, `city`, `cod_amount`, إلخ. عناصر الطلب تظهر لاحقاً عند التأكيد (في `confirmation_snapshot.items`) أو من واجهة الـ ERP مباشرة (/api/erp/drafts) إن استُخدمت. راجع `docs/call-center/GAPS-AND-TRANSFORMATION.md` § 2.7 و `erp-draft-dt-response-shape.md`.

## الإجراءات (في الواجهة)

- **اتصال** — متاح للطلبات ذات الحالة "جديد" أو "مجدول" (ما لم يكن الطلب مقفلاً أو في فترة انتظار).
- **عرض** — فتح تفاصيل الطلب / الجلسة.
- **تحويل للهاب** — متاح عندما تكون الحالة "مؤكد" (بعد موافقة القائد إن وُجدت).

تم تجميع كل الحالات أعلاه من: `OrdersTable.jsx` (getServiceTypeDisplay)، `CallSessionFAB.jsx` (getStatusLabel)، و`app/models/order.py` (OPEN_STATUSES وحالات الطلب).

| الاستخدام | Method | المسار | MCP `operation_id` |
